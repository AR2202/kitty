  Kitty Type Checker: Improvement Plan


  ---
  Completed

  The following items from the original plan have been implemented:

  - ToText now validates that its input is a convertible type (KInt, KFloat, KBool, KChar, KString)
  - ToNum now validates that its input is KString; dead unreachable fallthrough case removed
  - Letters now accepts any expression of type KString, not just string literals
  - Pop rewritten to reason about the result type rather than the AST shape;
    handles all list expressions, not just literals and bare variables
  - Push error message now correctly interpolates the existing list element type
  - UnwrapAs now narrows the unwrapped variable to wantedType inside the block
  - Suspicious unused imports removed (Control.Exception, Foreign.C, etc.)
  - Two remaining stale imports removed (void from Control.Monad, Data.Bits)
  - BuiltIn removed from KittyAST entirely
  - All error messages rewritten for a child audience
  - Function definition type checking implemented: parameters in scope, body type
    checked against declared return type, registered in _functionTypes on success
  - Function call type checking implemented: argument count and types checked,
    child-friendly error messages for all failure cases
  - 14 function type checker tests added to Spec.hs (all passing)

  ---
  Outstanding Issues

  ---
  1. unifyTypeEnvs workaround still in place — TypeChecker.hs (Medium)

  The current implementation:

    unifyTypeEnvs tenv1 tenv2 =
      tenv1
        { _varTypes =
            M.map (unwrapOneOfIfEqual . oneOfVoidIfMissing) $
              M.unionWith OneOf (_varTypes tenv1) (_varTypes tenv2)
        }

  M.unionWith OneOf creates OneOf t t when both branches assign the same type to
  a variable, and unwrapOneOfIfEqual then cleans it up. This works correctly but
  is convoluted. A cleaner approach avoids creating the bad OneOf in the first place:

    unifyTypeEnvs envA envB =
      TypeEnv
        { _functionTypes = Map.unionWith unifyFuns (_functionTypes envA) (_functionTypes envB)
        , _varTypes      = Map.unionWith unifyVars (_varTypes envA) (_varTypes envB)
        }
      where
        unifyVars a b
          | a == b    = a          -- same type: no wrapping needed
          | otherwise = OneOf a b  -- different: create union

  ---
  2. Double parsing in the REPL — REPL.hs (Medium)

  The interactive REPL path parses the input text twice: once inside updateTypeEnv
  and again inside parseReplT. The type checker and evaluator see independent parse
  results rather than the same AST object.

  Parsec is deterministic so both parses always produce the same AST, making this
  an architectural smell rather than a current bug. The fix is to parse once and
  thread the resulting AST through both the type checker and evaluator.

  This is structurally enforced by the Typed AST pattern (see design section below).

  ---
  3. evalFile command uses empty type environment — REPL.hs line 54 (Low)

  The evalFile command type-checks against initialTypeEnv (the empty environment),
  not the current session's tenv. A file cannot reference variables defined in the
  REPL session, and after the file runs, the REPL continues with its old state —
  nothing from the file persists.

  This may be intentional isolation, but is worth a conscious decision.

  ---
  4. Exported evaluator functions bypass type checking — Evaluator.hs (Low)

  Several exported functions skip the type checker entirely:

    parseEvalAndPrintEnv    -- Evaluator.hs:341
    parseEvalAndPrintResult -- Evaluator.hs:346
    parseRepl               -- Evaluator.hs:421

  None are called from the main REPL flow, but they are exported and could be
  called from tests or future code without realising they are unguarded. They
  should either be removed or have a type-check step added.

  ---
  Design notes — for reference

  ---
  Does adding functions require Hindley-Milner?

  Not necessarily — it depends on whether you require annotations.

  If function parameters have declared types, monomorphic functions are entirely
  tractable with the current architecture:

    define add(x: wholeNumber, y: wholeNumber) -> wholeNumber
      x + y
    enddefine

  Type-checking this requires no new machinery. Extend TypeEnv with the parameter
  types, check the body, verify the last expression matches the declared return type.

  Where you hit a wall is if you want to infer parameter types without annotations:

    define add(x, y)
      x + y     -- what are x and y?
    enddefine

  Here you see x + y and know both must be numeric and equal, but you can't determine
  which numeric type from the body alone. You'd need constraint solving — the core of HM.

  So: annotated monomorphic functions do not need HM. Unannotated functions do.

  ---
  When does HM become truly necessary?

  1. Polymorphic functions — the canonical HM use case.

    define identity(x)
      x
    endidentify

    With monomorphic inference, identity can only ever have one concrete type. With
    HM's let-generalisation, it gets the type ∀a. a → a. Without HM, you cannot
    write a generic identity, head, map, or filter.

  2. Type parameters on data structures

    define first(xs: List of T) -> T

    You need type variables (T here), requiring unification to check that the T in
    the argument and the T in the return type refer to the same thing.

  3. Type class constraints

    define sort(xs: List of T) -> List of T   -- T must be orderable

    This requires HM extended with constraints — roughly what Haskell's type system is.

  Short answer: annotated functions → no HM needed. Polymorphism → HM is the standard
  path. For Kitty targeting children, explicitly annotated functions are probably the
  right call, deferring HM for a long time.

  ---
  Alternative designs worth knowing about

  Bidirectional type checking

  Rather than one typeOf (inference only), two modes:

    infer :: KittyAST -> TypeEnv -> Either KittyError KType
    check :: KittyAST -> KType -> TypeEnv -> Either KittyError ()

  infer produces a type bottom-up (current approach). check takes an expected type
  and verifies the expression against it — flowing information top-down. The two
  modes call each other.

  Used by Rust, Swift, and Kotlin. Produces significantly better error messages than
  HM because you always know what type was expected at the point of failure. Requires
  annotations at certain boundaries (function parameters), which for a pedagogical
  language is arguably a feature — it makes types visible to learners.

  Typed AST

  Instead of typeOf returning a KType separately, the type checker transforms the AST
  into one where every node carries its type:

    data AST a
      = IntLit Int
      | Expr Operator (AST a) (AST a)
      | Variable String
      | ...
      deriving Functor

    type UntypedAST = AST ()
    type TypedAST   = AST KType

    typeCheck :: UntypedAST -> TypeEnv -> Either KittyError TypedAST

  Benefits for Kitty:
  - The evaluator can read the type off any node without re-running the type checker
  - It is impossible to accidentally pass an unchecked AST to the evaluator —
    the types enforce the pipeline and eliminate the double-parsing smell
  - Error messages become easier because you always have the type of every
    sub-expression to hand

  The cost is a heavier refactor: every AST case in the type checker has to construct
  a typed node rather than just return a KType. This is the direction production
  compilers (including GHC) go.

  Constraint-based inference (the HM implementation strategy)

  Separate the type checker into two phases:

    -- Phase 1: walk AST, emit constraints between type variables
    generate :: UntypedAST -> State FreshVars ([Constraint], TypeVar)

    -- Phase 2: solve constraints by unification
    solve :: [Constraint] -> Either KittyError Substitution

    -- Phase 3: apply substitution to get concrete types
    apply :: Substitution -> TypeVar -> KType

  A Constraint is an assertion that two types must be equal: TypeVar 3 ~ KInt.
  This architecture makes it easy to give good error messages and is how most HM
  implementations work in practice.

  ---
  Recommendation for Kitty

  1. ✓ Functions with explicit type annotations implemented. KittyTypes updated,
     type checker handles definitions and calls, 14 tests passing.
     Next: parser support for function definition syntax.

  2. Hold off on HM. Polymorphism is conceptually tricky for beginners anyway.

  3. Consider bidirectional checking as a medium-term upgrade for better error messages.

  4. Consider the Typed AST pattern when tidying the evaluator-type checker pipeline.
     It enforces that the evaluator only ever sees type-checked code and eliminates
     the double-parsing smell structurally.
